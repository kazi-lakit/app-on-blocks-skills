# blocks-deployment-readiness

Prepares a SELISE Blocks project for deployment by checking and generating required configuration files. This skill handles **codebase preparation only** — it does NOT call CloudBuild APIs or manage running deployments. For triggering/managing builds via API, use the `blocks-deployment` skill.

## Quick Start: Make a Project Deployment-Ready

See `flows/make-deployment-ready.md` for the full flow. High-level steps:

1. **Ask deployment method** — Traditional CI/CD (GitHub Actions) or Cloud Portal Direct?
2. **Detect project type** — see below
3. **Run readiness checks** (parallel) — env files, build config, Dockerfile, hosting config
4. **Report status** — show what's present vs missing
5. **Fix missing components** — generate from templates in `references/`
6. **Run build execution check** — `check-build-passes` verifies `npm run build:dev`, `npm run build:stg`, and `npm run build:prod` all pass
7. **Trigger deployment** — push to GitHub branch (dev → stg → prod pipeline for Path 1) or connect repo to cloud portal (Path 2)

### Project Type Detection

| Indicator | Framework | Dockerfile output path | Build command |
|-----------|-----------|----------------------|---------------|
| `package.json` + `next.config.ts` with `output: 'standalone'` | Next.js (standalone) | `.next/standalone/` | `next build` |
| `package.json` + `vite.config.ts` | Vite / React SPA | `/app/build` | `vite build` |
| `package.json` + `angular.json` | Angular | `/app/dist/<name>` | `ng build` |
| `pubspec.yaml` | Flutter Web | `/app/build/web` | `flutter build web` |
| `*.csproj` (Blazor WASM) | Blazor WebAssembly | `/app/publish/wwwroot` | `dotnet publish` |
| `*.csproj` (Blazor Server) | Blazor Server | (self-contained) | `dotnet run` |

## Deployment Method

Before generating any configuration, **ask the user which deployment method they want**.

### Path 1: Traditional CI/CD (Docker Hub → AKS)

The project builds Docker containers via GitHub Actions and pushes to Docker Hub. Credentials must be baked into the build at container build time.

**Use when:** You want full control over the build pipeline, or your organization requires it.

**Files needed:**
- `.env.dev`, `.env.stg`, `.env.prod` — with real credentials
- `Dockerfile` — multi-stage build (node builder + node+nginx runner for Next.js)
- `nginx.conf` — reverse proxy config
- `start.sh` — startup orchestrator (Next.js only)
- `next.config.ts` / `vite.config.ts` — build tool config

**Note:** GitHub workflow files are NOT generated or required by this skill. If workflows are needed, they should be managed separately by the infrastructure team.

### Path 2: Cloud Portal Direct

The project source is connected to the Blocks Cloud Portal, which handles the build and deployment. Credentials are injected by the portal at runtime — they do **not** need to be in the source code.

**Use when:** You want the simplest path — just push code and the portal deploys it.

**Files needed:**
- `.env.dev` — with real credentials (for local dev)
- `.env.stg` — with real credentials (for staging builds via portal)
- `.env.prod` — with placeholder values (portal injects real credentials at runtime)
- `Dockerfile` — standard multi-stage build
- `nginx.conf` — reverse proxy config
- `start.sh` — startup orchestrator (Next.js only)
- `next.config.ts` / `vite.config.ts` — build tool config

**Credentials:** Dev and staging need real values. Production uses placeholder values in `.env.prod` — the portal injects real credentials at runtime. Never commit real credentials to the codebase.

### Quick Decision Guide

| Question | Path 1: Traditional CI/CD | Path 2: Cloud Portal Direct |
|----------|------------------------|------------------------------|
| Who triggers builds? | GitHub Actions (push to branch) | Blocks Cloud Portal |
| Env files needed? | `.env.dev`, `.env.stg`, `.env.prod` | `.env.dev`, `.env.stg`, `.env.prod` (all 3; prod uses placeholders) |
| Prod credentials in source? | Yes — in `.env.prod` | No — injected by portal |
| GitHub workflows needed? | No — managed by infra team | No — portal handles CI/CD |
| When to use | Full pipeline control required | Simplest path, portal-managed |

> **Important:** If the user says "deploy via cloud portal" or "blocks cloud deployment" without mentioning GitHub workflows, assume **Path 2**. Do NOT generate `.env.prod` with real credentials in Path 2 — the portal handles production credentials.

### Canonical Dockerfile Pattern (Next.js Standalone)

```dockerfile
FROM node:21-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
ARG ci_build
RUN mkdir -p /app/log
COPY .env.${ci_build} .env.local
ENV NEXT_TELEMETRY_DISABLED=1
ENV NEXT_ENV=${ci_build}
RUN NODE_OPTIONS="--max-old-space-size=4096" npm run build:${ci_build}

FROM node:21-alpine AS runner
WORKDIR /app
RUN apk add --no-cache nginx
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1
ENV PORT=3000
ENV HOSTNAME="0.0.0.0"
RUN addgroup --system --gid 1001 nodejs \
    && adduser --system --uid 1001 nextjs
COPY --from=builder /app/public ./public
RUN mkdir -p .next && chown nextjs:nodejs .next
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
COPY nginx.conf /etc/nginx/http.d/default.conf
COPY start.sh ./start.sh
RUN chmod +x ./start.sh
EXPOSE 80
CMD ["./start.sh"]
```

Key: `npm run build:${ci_build}` (shell expansion, not plain `$ci_build`). The `ci_build` arg maps to `build:dev`, `build:stg`, or `build:prod` scripts. Each maps to a corresponding `.env.dev`, `.env.stg`, or `.env.prod` file.

### Canonical nginx.conf Pattern (Next.js Reverse Proxy)

```nginx
server {
  server_tokens off;
  client_max_body_size 200m;
  gzip             on;
  gzip_comp_level  6;
  gzip_min_length  1000;
  gzip_proxied     expired no-cache no-store private auth;
  gzip_types       text/plain application/x-javascript text/xml text/css application/xml text/javascript application/javascript application/json application/font-woff application/font-woff2 application/vnd.ms-fontobject application/x-font-ttf font/opentype;

  location / {
    proxy_pass http://localhost:3000;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection 'upgrade';
    proxy_set_header Host $host;
    proxy_cache_bypass $http_upgrade;
  }
}
```

Key: nginx proxies to `localhost:3000` where Next.js runs — NOT static file serving.

### Canonical start.sh Pattern

```sh
#!/bin/sh
set -e
node server.js &
sleep 2
nginx -g 'daemon off;'
```

### Canonical next.config.ts Pattern

```typescript
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: 'standalone',
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "images.unsplash.com",
      },
    ],
  },
};

export default nextConfig;
```

## Required Environment Variables

### Framework Env Prefix

Vite and Next.js use different prefixes for browser-exposed variables:

| Framework | Prefix | Example |
|-----------|--------|---------|
| Vite / React SPA | `VITE_` | `VITE_API_BASE_URL`, `VITE_X_BLOCKS_KEY` |
| Next.js | `NEXT_PUBLIC_` | `NEXT_PUBLIC_BLOCKS_API_URL`, `NEXT_PUBLIC_X_BLOCKS_KEY` |
| Angular | None (no browser prefix) | `API_BASE_URL`, `X_BLOCKS_KEY` (injected at build) |

> **Angular note:** Angular apps do not expose env vars to the browser at runtime. Use Angular environment files (`src/environments/environment.ts`) instead.

### Per-Framework Variable Names

**Next.js (use `NEXT_PUBLIC_` prefix):**

| Variable | Required | Description |
|----------|----------|-------------|
| `NEXT_PUBLIC_BLOCKS_API_URL` | Yes | API endpoint |
| `NEXT_PUBLIC_X_BLOCKS_KEY` | Yes | Blocks project key |
| `NEXT_PUBLIC_BLOCKS_OIDC_CLIENT_ID` | No | OIDC client for auth |
| `NEXT_PUBLIC_BLOCKS_OIDC_REDIRECT_URI` | No | OIDC redirect URI |

**Vite / React SPA:**

| Variable | Required | Description |
|----------|----------|-------------|
| `VITE_API_BASE_URL` | Yes | API endpoint (dev/stg/prod) |
| `VITE_X_BLOCKS_KEY` | Yes | Blocks project key |
| `VITE_PROJECT_SLUG` | Yes | Project identifier |
| `VITE_BLOCKS_OIDC_CLIENT_ID` | Yes | OIDC client for auth |
| `VITE_BLOCKS_OIDC_REDIRECT_URI` | Yes | OIDC redirect URI (must match deployed domain) |

### Next.js Build Scripts

```json
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "build:dev": "NEXT_ENV=development next build",
    "build:stg": "NEXT_ENV=staging next build",
    "build:prod": "NEXT_ENV=production next build",
    "start": "next start",
    "lint": "eslint"
  }
}
```

Key: `NEXT_ENV` is set to `development`, `staging`, or `production`. The `ci_build` arg in the Dockerfile maps to `build:dev`, `build:stg`, or `build:prod` — each calling a corresponding `.env.dev`, `.env.stg`, or `.env.prod` file. The `NEXT_ENV` value (`development`/`staging`/`production`) is used internally by Next.js, while the build script name (`build:dev`/`build:stg`/`build:prod`) determines which env file is loaded.

## Common Pitfalls

> [!WARNING]
>
> - **Never ask users to share credentials with AI** — Use placeholder values in templates. Never generate files with real API keys or passwords.
>
> - **Generate templates, not real values** — When creating env files, use placeholder comments like `# NEXT_PUBLIC_X_BLOCKS_KEY = <your-project-key>`. Never embed actual credentials.
>
> - **Don't assume all env vars are present** — Check each env file individually. `.env.dev`, `.env.stg`, and `.env.prod` may have different variable coverage.
>
> - **Path 2 (Cloud Portal Direct): always create .env.prod with placeholders** — The portal injects real credentials at runtime, but `.env.prod` should still exist as a template. Use placeholder values like `<your-project-key>` — never real credentials in the codebase.
>
> - **Use correct env prefix per framework** — Vite apps use `VITE_*`, Next.js uses `NEXT_PUBLIC_*`. Using the wrong prefix means the variable is `undefined` at runtime.
>
> - **Generate all three env files** — Always create `.env.dev`, `.env.stg`, and `.env.prod`. Skipping staging means the CI/CD pipeline has no way to build a staging environment.
>
> - **Verify build script names match CI workflow** — The `ci_build` arg in the Dockerfile maps to `build:dev`, `build:stg`, or `build:prod` scripts in package.json. Mismatched names cause ACR build failures.
>
> - **Next.js: detect standalone pattern first** — Check `next.config.ts` for `output: 'standalone'` before applying the Next.js standalone template. Without it, use the Vite static fallback.
>
> - **Don't trigger deployment for the user** — This skill prepares code only. Triggering builds is handled by `blocks-deployment` or the infrastructure team.

- [ ] Two-stage build: `FROM node:21-alpine AS builder` → `FROM nginx:stable-alpine`
- [ ] `ARG ci_build` defined and `npm run build:${ci_build}` called

## Troubleshooting

### Build Failures

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Build fails | `ci_build` arg doesn't match script name | Use `npm run build:${ci_build}` — valid values are `dev`, `stg`, `prod` (note `${}` syntax, not `$ci_build`) |
| Build fails | Wrong output path | Verify `.next/standalone/` for Next.js, `/app/build` for Vite |
| Build fails | Memory limit exceeded | Add `NODE_OPTIONS="--max-old-space-size=4096"` |


### Environment Variable Issues

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Env vars not loaded (Next.js) | `COPY .env.${ci_build} .env.local` missing in Dockerfile | Add the COPY directive |
| `NEXT_PUBLIC_*` vars undefined at runtime | Using wrong prefix | Replace `VITE_*` with `NEXT_PUBLIC_*` for browser-exposed vars |
| `VITE_*` used in Next.js project | Wrong prefix for framework | Use `NEXT_PUBLIC_*` for Next.js — `VITE_*` is ignored |
| Env vars not loaded (Vite) | Missing `envPrefix: 'VITE_'` in vite.config.ts | Add `envPrefix: 'VITE_'` to Vite config |
| Missing OIDC redirect | `*_BLOCKS_OIDC_REDIRECT_URI` absent | Add it — OIDC login fails silently without it |

### nginx/Next.js Issues

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| 404 on all routes (Next.js) | nginx not proxying to Next.js | Verify `proxy_pass http://localhost:3000;` in nginx.conf |
| 404 on non-root routes (Vite) | Missing SPA fallback | Add `try_files $uri $uri/ /index.html;` |
| Container exits immediately | nginx not in foreground | Ensure `nginx -g 'daemon off;'` is used, not `nginx` alone |
| File upload too large | Default nginx body size | Add `client_max_body_size 200m;` to nginx.conf |
| Slow page loads | Gzip not enabled | Add `gzip on;` to nginx.conf |

### Deployment Issues

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Container build fails | Dockerfile has errors | Verify multi-stage build syntax |
| Environment vars not loaded | Wrong env file copy directive | Check `COPY .env.${ci_build} .env.local` for Next.js |
| OIDC login fails | Wrong redirect URI | Check `*_BLOCKS_OIDC_REDIRECT_URI` matches domain |
| Build output not found | Wrong output path | Verify Dockerfile COPY path matches build output |

### Credential/Security Issues

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Deployment requires credentials | User entered values manually | Use Blocks Cloud Portal for credential management |
| API key exposed in file | Real credentials in template | Regenerate env files with placeholder `<your-project-key>` |
| Prod credentials in source | `.env.prod` has real values | Use placeholder values — portal injects real credentials at runtime |

### Angular/Next.js Issues

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Build output not found | Wrong dist path | Change `/app/build` to `/app/dist/<project-name>` in Dockerfile |
| Angular build fails | Missing Angular CLI | Dockerfile builder stage needs `npm install` before build |
| Next.js routes 404 | Next.js standalone not configured | Ensure `output: 'standalone'` in `next.config.ts` |
| Next.js container exits | `start.sh` not executable | Add `RUN chmod +x ./start.sh` to Dockerfile |
